## <a name="understanding-iactionconstraint"></a><span data-ttu-id="9a707-896">IActionConstraint 이해</span><span class="sxs-lookup"><span data-stu-id="9a707-896">Understanding IActionConstraint</span></span>

> [!NOTE]
> <span data-ttu-id="9a707-897">이 섹션에서는 프레임워크의 내부 구조 및 MVC가 실행할 작업을 선택하는 원리를 자세히 알아봅니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-897">This section is a deep-dive on framework internals and how MVC chooses an action to execute.</span></span> <span data-ttu-id="9a707-898">일반적인 응용 프로그램에는 사용자 지정 `IActionConstraint`가 필요 없습니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-898">A typical application won't need a custom `IActionConstraint`</span></span>

<span data-ttu-id="9a707-899">이 인터페이스에 익숙하지 않은 분들도 이미 `IActionConstraint`를 사용해 보셨을 것입니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-899">You have likely already used `IActionConstraint` even if you're not familiar with the interface.</span></span> <span data-ttu-id="9a707-900">`[HttpGet]` 특성 그리고 그와 비슷한 `[Http-VERB]` 특성들은 `IActionConstraint`를 구현하여 작업 메서드 실행을 제한합니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-900">The `[HttpGet]` Attribute and similar `[Http-VERB]` attributes implement `IActionConstraint` in order to limit the execution of an action method.</span></span>

```csharp
    public IActionResult Edit() { }

    public IActionResult Edit(...) { }
```

<span data-ttu-id="9a707-901">기본 규칙 기반 경로를 사용한다고 가정할 때, URL 경로 `/Products/Edit`는 여기에 보이는 두 작업 **모두**와 매칭하는 `{ controller = Products, action = Edit }` 값을 생성합니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-901">Assuming the default conventional route, the URL path `/Products/Edit` would produce the values `{ controller = Products, action = Edit }`, which would match **both** of the actions shown here.</span></span> <span data-ttu-id="9a707-902">`IActionConstraint` 용어에서는 두 작업이 경로 데이터와 매칭되므로 두 작업이 후보로 간주된다고 말할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-902">In `IActionConstraint` terminology we would say that both of these actions are considered candidates - as they both match the route data.</span></span>

<span data-ttu-id="9a707-903">`HttpGetAttribute`가 실행되면 *Edit()* 는 *GET*에 대해서만 일치하고 다른 HTTP 동사에 대해서는 일치하지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-903">When the `HttpGetAttribute` executes, it will say that *Edit()* is a match for *GET* and isn't a match for any other HTTP verb.</span></span> <span data-ttu-id="9a707-904">`Edit(...)` 작업은 정의된 제약 조건이 없으므로 모든 HTTP 동사와 매칭됩니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-904">The `Edit(...)` action doesn't have any constraints defined, and so will match any HTTP verb.</span></span> <span data-ttu-id="9a707-905">따라서 `POST`를 고려해본다면 `Edit(...)`만 일치합니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-905">So assuming a `POST` - only `Edit(...)` matches.</span></span> <span data-ttu-id="9a707-906">하지만 `GET`의 경우에는 여전히 두 작업 모두 매칭될 수 있습니다. 그러나`IActionConstraint`가 적용된 작업이 적용되지 않은 작업보다 항상 *더 적합한* 것으로 간주됩니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-906">But, for a `GET` both actions can still match - however, an action with an `IActionConstraint` is always considered *better* than an action without.</span></span> <span data-ttu-id="9a707-907">따라서 `Edit()`에는 `[HttpGet]`가 있으므로 보다 구체적인 것으로 간주되며, 두 작업이 모두 매칭될 수 있는 경우에도 이 작업이 선택됩니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-907">So because `Edit()` has `[HttpGet]` it's considered more specific, and will be selected if both actions can match.</span></span>

<span data-ttu-id="9a707-908">개념적으로 `IActionConstraint`는 *오버로딩*의 한 가지 형태이지만, 같은 이름의 메서드를 오버로딩하는 대신, 동일한 URL을 매칭하는 작업 간에 오버로딩합니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-908">Conceptually, `IActionConstraint` is a form of *overloading*, but instead of overloading methods with the same name, it's overloading between actions that match the same URL.</span></span> <span data-ttu-id="9a707-909">특성 라우팅에서도 `IActionConstraint`를 사용하며, 다른 두 컨트롤러의 작업이 모두 후보로 간주될 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="9a707-909">Attribute routing also uses `IActionConstraint` and can result in actions from different controllers both being considered candidates.</span></span>

### <a name="implementing-iactionconstraint"></a><span data-ttu-id="9a707-910">IActionConstraint 구현</span><span class="sxs-lookup"><span data-stu-id="9a707-910">Implementing IActionConstraint</span></span>

```csharp
public class CountrySpecificAttribute : Attribute, IActionConstraint
{
    private readonly string _countryCode;

    public CountrySpecificAttribute(string countryCode)
    {
        _countryCode = countryCode;
    }

    public int Order
    {
        get
        {
            return 0;
        }
    }

    public bool Accept(ActionConstraintContext context)
    {
        return string.Equals(
            context.RouteContext.RouteData.Values["country"].ToString(),
            _countryCode,
            StringComparison.OrdinalIgnoreCase);
    }
}
```

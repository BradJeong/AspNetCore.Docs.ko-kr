<span data-ttu-id="dba63-420">동시성 위반 기회가 여러 수 있는 모든 응용 프로그램에 존재 하는 동시 사용자 데이터를 업데이트 또는 삭제 합니다.</span><span class="sxs-lookup"><span data-stu-id="dba63-420">Opportunities for concurrency violations exist in every application that allows multiple, concurrent users to update or delete data.</span></span> <span data-ttu-id="dba63-421">경우 이러한 위반은 두 명의 사용자가 누구 든 지 가져옵니다에서 마지막 쓰기 "wins"에 있는 동일한 데이터를 동시에 업데이트 변경 내용을 변경 하면 다른 사용자의 덮어쓰지에 대해서 고려 하지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="dba63-421">If such violations are not accounted for, when two users simultaneously update the same data whoever gets in the last write "wins," overwriting the other user's changes changes.</span></span> <span data-ttu-id="dba63-422">또는 개발자가 어느 낙관적 또는 비관적 동시성 제어를 구현할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="dba63-422">Alternatively, developers can implement either optimistic or pessimistic concurrency control.</span></span> <span data-ttu-id="dba63-423">낙관적 동시성 제어 가정 동시성 위반 빈번 하지 않습니다 및 단순히 업데이트를 허용 하지 않는 명령에 해당 합니다 동시성 위반을 삭제 합니다.</span><span class="sxs-lookup"><span data-stu-id="dba63-423">Optimistic concurrency control assumes that concurrency violations are infrequent and simply disallows an update or delete command that would constitute a concurrency violation.</span></span> <span data-ttu-id="dba63-424">비관적 동시성 제어는 동시성 위반 빈도가 및 단순히 한 사용자의 거부 된 업데이트 또는 삭제 명령 허용 되지 않는 가정 합니다.</span><span class="sxs-lookup"><span data-stu-id="dba63-424">Pessimistic concurrency control assumes that concurrency violations are frequent and simply rejecting one user's update or delete command is not acceptable.</span></span> <span data-ttu-id="dba63-425">비관적 동시성 제어와 레코드 업데이트 잠가, 다른 사용자가을 수정 하거나 잠겨 있는 동안에 레코드를 삭제 하면 것을 방지 합니다.</span><span class="sxs-lookup"><span data-stu-id="dba63-425">With pessimistic concurrency control, updating a record involves locking it, thereby preventing any other users from modifying or deleting the record while it is locked.</span></span>
